PROJECT: Team Management (RBAC)

INTRODUCTION

This repository is a role-based access control (RBAC) team and task management system.
The architecture is a two-tier app:
- server/: Express.js API (Node, Mongoose) ‚Äî responsible for authentication, RBAC logic, SSE realtime notifications, and persistence.
- client/: Vite + React SPA ‚Äî user interface, consumes API, manages session state and real-time sync via SSE.

High-level goals
- Let admins/managers manage teams, roles, and memberships.
- Allow users to be members of multiple teams with per-team roles.
- Keep client UI in-sync when role/membership changes (SSE).

AUTHENTICATION & AUTH FLOW (end-to-end)

1) Registration / Login
- Client calls `POST /api/auth/register` or `POST /api/auth/login` (see: server/src/features/auth.js).
- Server validates, creates user or checks password hash (bcryptjs), and issues a JWT via `jwt.sign`.
- Token payload contains `sub` (user id), `email`, and boolean flags `isAdmin` and `isManager` (derived from email rules) for quick UI gating.

2) Token storage & usage
- Client stores token in `localStorage` under `rbac_token` (client/src/state/AuthContext.jsx and client/src/api/api.js).
- Axios `api` instance attaches `Authorization: Bearer <token>` header to requests.

3) Protected routes
- Server middleware `requireAuth` (server/src/middleware/auth.js) extracts token from `Authorization` header or `?token=` query (used for SSE), verifies with `jwt.verify`, sets `req.authUser`, or throws 401.
- Higher-level guards `requireAdmin` and `requirePrivileged` (server/src/middleware/access.js) use email-based privileged rules + membership-based checks to authorize functionality.

4) Expanded user info
- Client calls `GET /api/auth/me` to obtain membership-aware `user` information. Server computes dynamic `isAdmin`/`isManager` by querying `Membership` and populating `roleId`.

REALTIME (SSE) OVERVIEW

- Server exposes `/api/realtime/stream` (server/src/features/realtime.js) ‚Äî an SSE endpoint that registers a response as a subscriber for a user id.
- `server/src/shared/realtime.js` manages an in-memory map of subscribers and provides `emitRoleChange(...)` to notify connected clients.
- Client `AuthContext` (client/src/state/AuthContext.jsx) connects via `new EventSource(streamUrl)` and listens for `role-change` events; on receipt it re-fetches `/api/auth/me` to refresh local user state.

SECURITY NOTES & RECOMMENDATIONS

- Password hashing: project uses `bcryptjs` (pure JS). For production prefer native `bcrypt` for performance.
- Token storage: storing JWT in `localStorage` exposes it to XSS. For stronger security use HTTP-only secure cookies and short-lived access tokens + refresh tokens.
- SSE auth: current SSE connects with `?token=` because `EventSource` cannot set headers. This leaks tokens to logs and referers. Recommended fixes:
  - Use HTTP-only cookies for SSE authentication, or
  - Issue short-lived one-time stream tokens stored server-side and passed as query param, revoking them after use.
- Token revocation: JWTs are stateless. If you need immediate logout/invalidation add a blacklist (Redis) or token versioning on user records.

PERFORMANCE NOTES

- Membership lookups in `getCurrentUserFromDb` use `Membership.find(...).populate('roleId')`. If memberships grow large, optimize by using `exists()` queries or aggregations to compute boolean flags rather than full populate.
- Bcrypt compare may be CPU heavy under load; native `bcrypt` or worker/offload is recommended for production.

TROUBLESHOOTING CHECKLIST

- If login is slow: check server logs for timing (server/src/features/auth.js includes timing log points), check bcrypt CPU usage, and check DB latency.
- If SSE not connecting: verify `API_BASE_URL` and that `/realtime/stream` is reachable; check browser console for EventSource errors.
- If permissions appear incorrect: call `/api/auth/me` and inspect `user` object and `memberships` returned by the server.
